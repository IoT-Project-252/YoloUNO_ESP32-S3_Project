\subsection{Task 1 -- Điều khiển LED đơn theo điều kiện nhiệt độ}

Task 1 mở rộng chức năng LED đơn của dự án gốc bằng cách liên kết hành vi nhấp nháy với dữ liệu nhiệt độ đo được theo thời gian thực từ cảm biến DHT20. Ba mức hành vi khác nhau -- mỗi mức ứng với một khoảng nhiệt độ cụ thể -- được điều chỉnh cả về độ sáng và tần suất nhấp nháy thông qua phần cứng PWM. Cơ chế đồng bộ hóa giữa task đọc cảm biến và task điều khiển LED được thực hiện hoàn toàn qua \textbf{FreeRTOS binary semaphore}, không tạo thêm biến toàn cục mới theo đúng yêu cầu đề bài.

\subsubsection*{Mục tiêu hiện thực}

Nhóm hiện thực Task 1 theo hướng tách rõ chức năng thành hai task FreeRTOS độc lập:
\begin{itemize}
    \item \texttt{temp\_humi}: đọc DHT20 định kỳ và phát tín hiệu trạng thái nhiệt độ qua semaphore.
    \item \texttt{led\_blinky}: nhận tín hiệu trạng thái và điều khiển LED GPIO 48 theo hành vi tương ứng.
\end{itemize}

Hai task này được tạo trong \texttt{setup()} của \texttt{main.cpp}:
\begin{lstlisting}[language=c, caption=Khởi tạo các task liên quan Task 1 trong main.cpp]
xTaskCreate(temp_humi,  "Task Read Temperature & Humidity",
            4096, projectSharedData, 1, NULL);
xTaskCreate(led_blinky, "Task LED Blinky",
            2048, projectSharedData, 1, NULL);
\end{lstlisting}

\subsubsection*{Định nghĩa ngưỡng nhiệt độ và hành vi LED}

Đề bài yêu cầu ít nhất ba hành vi nhấp nháy tương ứng với ba khoảng nhiệt độ ($< 26\,^\circ\mathrm{C}$; $[26, 28]\,^\circ\mathrm{C}$; $> 28\,^\circ\mathrm{C}$). Nhóm triển khai bằng cách kết hợp điều chỉnh duty cycle PWM và chu kỳ bật/tắt để tạo ra sự phân biệt trực quan đa chiều:

\begin{table}[h!]
\centering
\caption{Ngưỡng nhiệt độ và hành vi LED của Task 1}
\label{tab:task1-thresholds}
\begin{tabular}{|c|c|c|c|c|}
\hline
\textbf{Vùng} & \textbf{Điều kiện} & \textbf{Duty cycle} & \textbf{Thời gian bật} & \textbf{Thời gian tắt} \\
\hline
Normal   & $T < 26\,^\circ\mathrm{C}$              & 64/255 $\approx$ 25\% & 1000 ms & 1000 ms \\
\hline
Warning  & $26 \leq T \leq 28\,^\circ\mathrm{C}$  & 160/255 $\approx$ 63\% & 500 ms & 500 ms \\
\hline
Critical & $T > 28\,^\circ\mathrm{C}$              & 255/255 = 100\%        & 200 ms & 200 ms \\
\hline
\end{tabular}
\end{table}

Việc sử dụng PWM hardware (khối LEDC của ESP32-S3) thay vì chỉ bật/tắt GPIO đơn giản mang lại ưu điểm: tần số sóng mang 5\,kHz hoàn toàn do phần cứng duy trì trong suốt chu kỳ \texttt{vTaskDelay()}, tách biệt hoàn toàn giữa tần số carrier và chu kỳ nhấp nháy do phần mềm điều khiển.

\subsubsection*{Cơ chế đồng bộ bằng semaphore -- mô hình producer-consumer}

Nhằm tuân thủ yêu cầu ``dùng biến từ task khác, không tạo biến mới'', nhóm thiết kế bộ ba binary semaphore chuyên dụng cho Task 1, khai báo trong cấu trúc \texttt{SharedData}:

\begin{lstlisting}[language=c, caption=Các semaphore dành riêng cho Task 1 (trích global.h)]
struct SharedData {
    // ...
    // Semaphore dieu kien nhiet do cho Task 1 (LED blinky)
    SemaphoreHandle_t semTempNormal;    // T < 26 do C
    SemaphoreHandle_t semTempWarning;   // 26 <= T <= 28 do C
    SemaphoreHandle_t semTempCritical;  // T > 28 do C
    // ...
};
\end{lstlisting}

Ba semaphore này được khởi tạo bằng \texttt{xSemaphoreCreateBinary()} trong \texttt{initGlobalData()} trước khi bất kỳ task nào được tạo. Theo đặc tính của FreeRTOS, binary semaphore khởi tạo ở trạng thái \textit{taken} (rỗng), vì vậy \texttt{led\_blinky} sẽ giữ nguyên \texttt{mode = 1} (Warning) cho đến khi lần đọc cảm biến đầu tiên hoàn thành.

\paragraph{Producer: task \texttt{temp\_humi}}
Sau mỗi chu kỳ đọc DHT20 thành công (mỗi 5\,000\,ms), task so sánh nhiệt độ với ngưỡng và \texttt{xSemaphoreGive()} đúng một semaphore tương ứng:

\begin{lstlisting}[language=c, caption=Phát semaphore nhiệt độ cho Task 1 (trích temp\_humi.cpp)]
// Task 1: nguong nhiet do cho LED blinky (<26, 26-28, >28)
if (t < 26.0f) {
    xSemaphoreGive(sharedData->semTempNormal);
} else if (t <= 28.0f) {
    xSemaphoreGive(sharedData->semTempWarning);
} else {
    xSemaphoreGive(sharedData->semTempCritical);
}
\end{lstlisting}

Điểm quan trọng: mỗi chu kỳ chỉ phát \textit{đúng một} semaphore, đảm bảo consumer không bao giờ nhận hai tín hiệu mâu thuẫn trong cùng một vòng lặp.

\paragraph{Consumer: task \texttt{led\_blinky}}
Task sử dụng \texttt{xSemaphoreTake} với \texttt{timeout = 0} (non-blocking) để thăm dò ba semaphore mỗi đầu chu kỳ nhấp nháy:

\begin{lstlisting}[language=c, caption=Vòng lặp điều khiển LED theo tín hiệu semaphore (trích led\_blinky.cpp)]
uint8_t mode = 1; // mac dinh: Warning (26-28 do C)

while (1) {
    // Non-blocking poll -- cap nhat mode neu co tin hieu moi
    if (xSemaphoreTake(sharedData->semTempNormal,   0) == pdTRUE) mode = 0;
    else if (xSemaphoreTake(sharedData->semTempWarning,  0) == pdTRUE) mode = 1;
    else if (xSemaphoreTake(sharedData->semTempCritical, 0) == pdTRUE) mode = 2;

    // Chon duty va thoi gian theo mode
    uint8_t  duty   = (mode == 0) ?  64 : (mode == 1) ? 160 : 255;
    uint32_t on_ms  = (mode == 0) ? 1000 : (mode == 1) ? 500 : 200;
    uint32_t off_ms = on_ms;

    ledcWrite(LED_PWM_CH, duty);           // Bat LED
    vTaskDelay(pdMS_TO_TICKS(on_ms));
    ledcWrite(LED_PWM_CH, 0);             // Tat LED
    vTaskDelay(pdMS_TO_TICKS(off_ms));
}
\end{lstlisting}

Cơ chế non-blocking đảm bảo LED không bao giờ bị đóng băng: nếu cảm biến chưa có đọc mới (ví dụ lỗi I2C tạm thời), \texttt{mode} giữ nguyên giá trị trước và LED tiếp tục nhấp nháy theo hành vi đã biết.

\subsubsection*{Cấu hình PWM phần cứng}

\begin{lstlisting}[language=c, caption=Khởi tạo LEDC PWM cho LED đơn (trích led\_blinky.cpp)]
static const uint8_t  LED_PWM_CH   = 0;    // LEDC channel 0
static const uint32_t LED_PWM_FREQ = 5000; // 5 kHz -- tren nguong nhan nhai mat nguoi
static const uint8_t  LED_PWM_RES  = 8;    // 8-bit: pham vi duty 0..255

ledcSetup(LED_PWM_CH, LED_PWM_FREQ, LED_PWM_RES);
ledcAttachPin(LED_GPIO, LED_PWM_CH); // GPIO 48 -- LED tich hop YoloUNO
\end{lstlisting}

Tần số 5\,kHz đảm bảo ánh sáng LED mượt mà, không rung, đồng thời phân giải 8-bit cung cấp 256 mức sáng -- đủ để tạo độ phân biệt trực quan rõ ràng giữa ba vùng nhiệt độ.

\subsubsection*{Quan hệ với các task khác -- tách semaphore theo consumer}

Task 1 dùng bộ semaphore \texttt{semTempNormal/Warning/Critical} riêng biệt, \textit{không dùng chung} với bộ semaphore \texttt{semNormal/Warning/Critical} phục vụ Task 3 (hiển thị LCD). Cả hai bộ đều được phát bởi cùng một producer là \texttt{temp\_humi}, nhưng với định nghĩa ngưỡng khác nhau:

\begin{table}[h!]
\centering
\caption{So sánh bộ semaphore Task 1 và Task 3 -- cùng producer, khác ngưỡng}
\label{tab:task1-vs-task3-sem}
\begin{tabular}{|c|c|c|}
\hline
\textbf{Semaphore} & \textbf{Ngưỡng phát} & \textbf{Consumer} \\
\hline
\texttt{semTempNormal}   & $T < 26\,^\circ\mathrm{C}$               & \texttt{led\_blinky} (Task 1) \\
\texttt{semTempWarning}  & $26 \leq T \leq 28\,^\circ\mathrm{C}$   & \texttt{led\_blinky} (Task 1) \\
\texttt{semTempCritical} & $T > 28\,^\circ\mathrm{C}$               & \texttt{led\_blinky} (Task 1) \\
\hline
\texttt{semNormal}   & $20 \leq T < 30\,^\circ\mathrm{C}$           & \texttt{displayLCD} (Task 3) \\
\texttt{semWarning}  & $30 \leq T < 38$ hoặc $7 \leq T < 20\,^\circ\mathrm{C}$ & \texttt{displayLCD} (Task 3) \\
\texttt{semCritical} & $T < 7$ hoặc $T \geq 38\,^\circ\mathrm{C}$  & \texttt{displayLCD} (Task 3) \\
\hline
\end{tabular}
\end{table}

Thiết kế này ngăn chặn tình trạng một consumer ``ăn cắp'' tín hiệu của consumer kia, đảm bảo Task 1 và Task 3 hoạt động hoàn toàn độc lập dù cùng đọc dữ liệu từ một nguồn.

\subsubsection*{Điểm nhấn kỹ thuật của Task 1}

\begin{itemize}
    \item \textbf{Giao tiếp không biến toàn cục}: toàn bộ thông tin trạng thái nhiệt độ được truyền qua semaphore -- \texttt{mode} là biến cục bộ trong stack của \texttt{led\_blinky}, không tồn tại trong vùng toàn cục.
    \item \textbf{Khả năng chịu lỗi}: cơ chế non-blocking trytake giúp LED duy trì hành vi cuối cùng khi cảm biến tạm thời không đọc được, không làm treo task.
    \item \textbf{Tách biệt hoàn toàn}: Task 1 không trực tiếp đọc biến \texttt{temperature} trong \texttt{SharedData}, mà chỉ nhận tín hiệu từ semaphore -- đảm bảo không cần thêm mutex bảo vệ dữ liệu đo.
    \item \textbf{PWM phần cứng}: duty cycle do LEDC hardware duy trì độc lập với vòng lặp phần mềm, tách biệt carrier frequency và blink period.
\end{itemize}

\subsubsection*{Tổng kết Task 1}

Task 1 đã đáp ứng đầy đủ yêu cầu của đề bài: có ít nhất ba hành vi nhấp nháy tương ứng ba khoảng nhiệt độ, sử dụng semaphore để đồng bộ giữa task cảm biến và task LED, và không tạo thêm biến toàn cục mới. Kết quả là hệ thống có một kênh cảnh báo trực quan liên tục -- LED nhấp nháy với cường độ và tần suất tăng dần theo nhiệt độ -- hoạt động song song và không xung đột với các task còn lại của hệ thống.

\subsection{Task 3 - Theo dõi nhiệt độ và độ ẩm với màn hình LCD}

Task 3 tập trung vào ba yêu cầu cốt lõi của đề bài: (1) đo nhiệt độ/độ ẩm theo chu kỳ, (2) hiển thị ít nhất 3 trạng thái trên LCD dựa theo ngưỡng đo, và (3) loại bỏ các biến toàn cục rời rạc bằng mô hình dữ liệu dùng chung có đồng bộ trong FreeRTOS.

\subsubsection*{Mục tiêu hiện thực}

Nhóm hiện thực Task 3 theo hướng tách rõ chức năng thành hai task FreeRTOS độc lập:
\begin{itemize}
    \item \texttt{temp\_humi}: đọc DHT20 và cập nhật dữ liệu đo vào vùng dùng chung.
    \item \texttt{displayLCD}: đọc trạng thái từ semaphore và render nội dung LCD tương ứng.
\end{itemize}

Hai task này được tạo trong \texttt{setup()} của \texttt{main.cpp}:
\begin{lstlisting}[language=c, caption=Khởi tạo các task liên quan Task 3 trong main.cpp]
xTaskCreate(temp_humi,  "TempHumi",   4096, projectSharedData, 1, NULL);
xTaskCreate(displayLCD, "DisplayLCD", 4096, projectSharedData, 1, NULL);
\end{lstlisting}

\subsubsection*{Tổ chức dữ liệu chia sẻ và loại bỏ global variables}

Thay vì dùng nhiều biến toàn cục rời rạc, hệ thống gom toàn bộ trạng thái dùng chung vào \texttt{SharedData}. Với Task 3, các thành phần quan trọng gồm:
\begin{itemize}
    \item \texttt{temperature}, \texttt{humidity}: dữ liệu đo mới nhất.
    \item \texttt{mutex}: bảo vệ truy cập dữ liệu đo (tránh race condition).
    \item \texttt{i2cMutex}: bảo vệ bus I2C dùng chung giữa DHT20 và LCD.
    \item \texttt{semNormal}, \texttt{semWarning}, \texttt{semCritical}: tín hiệu trạng thái hiển thị.
\end{itemize}

\begin{lstlisting}[language=c, caption=Cấu trúc SharedData dùng cho Task 3 (trích global.h)]
struct SharedData
{
    float temperature;
    float humidity;

    SemaphoreHandle_t mutex;
    SemaphoreHandle_t i2cMutex;

    SemaphoreHandle_t semNormal;
    SemaphoreHandle_t semWarning;
    SemaphoreHandle_t semCritical;
};
\end{lstlisting}

Cách tổ chức này đảm bảo dữ liệu được quản lý tập trung, giảm phụ thuộc chéo giữa module và phù hợp yêu cầu ``remove ALL global variables'' theo tinh thần đề bài.

\subsubsection*{Task đọc cảm biến DHT20 và phát tín hiệu trạng thái}

Task \texttt{temp\_humi} thực hiện ba bước chính trong mỗi chu kỳ:
\begin{enumerate}
    \item Khóa \texttt{i2cMutex} để đọc DHT20 an toàn.
    \item Khóa \texttt{mutex} để ghi \texttt{temperature/humidity} vào vùng chia sẻ.
    \item So sánh ngưỡng nhiệt độ và \texttt{xSemaphoreGive()} semaphore trạng thái tương ứng.
\end{enumerate}

Đoạn logic ngưỡng:
\begin{lstlisting}[language=c, caption=Phát semaphore theo ngưỡng nhiệt độ (trích temp\_humi.cpp)]
if (t >= 20 && t < 30)
{
    xSemaphoreGive(sharedData->semNormal);
}
else if ((t >= 30.0 && t < 38.0) || (t >= 7 && t < 20))
{
    xSemaphoreGive(sharedData->semWarning);
}
else
{
    xSemaphoreGive(sharedData->semCritical);
}
\end{lstlisting}

Tương ứng, hệ thống có ba mức trạng thái hiển thị như Bảng~\ref{tab:task3-thresholds}.

\begin{table}[h!]
\centering
\caption{Ngưỡng trạng thái hiển thị của Task 3}
\label{tab:task3-thresholds}
\begin{tabular}{|c|c|c|}
\hline
\textbf{Khoảng nhiệt độ} & \textbf{Trạng thái} & \textbf{Semaphore phát} \\
\hline
$20 \leq T < 30$ & NORMAL & \texttt{semNormal} \\
\hline
$30 \leq T < 38$ hoặc $7 \leq T < 20$ & WARNING & \texttt{semWarning} \\
\hline
$T < 7$ hoặc $T \geq 38$ & CRITICAL & \texttt{semCritical} \\
\hline
\end{tabular}
\end{table}

\subsubsection*{Task hiển thị LCD theo mô hình state machine}

Task \texttt{displayLCD} hoạt động như một state machine dựa trên semaphore. Ở mỗi vòng lặp:
\begin{itemize}
    \item Task kiểm tra lần lượt \texttt{semNormal}, \texttt{semWarning}, \texttt{semCritical} theo cơ chế non-blocking.
    \item Khi nhận được semaphore, task khóa \texttt{mutex} để lấy bản sao dữ liệu đo.
    \item Task khóa \texttt{i2cMutex} để render nội dung lên LCD, tránh tranh chấp bus I2C với task cảm biến.
\end{itemize}

\begin{lstlisting}[language=c, caption=Khối xử lý trạng thái WARNING trên LCD (trích display\_lcd.cpp)]
else if (xSemaphoreTake(sharedData->semWarning, 0) == pdTRUE)
{
    if (xSemaphoreTake(sharedData->mutex, pdMS_TO_TICKS(50)) == pdTRUE)
    {
        localTemp = sharedData->temperature;
        localHum = sharedData->humidity;
        xSemaphoreGive(sharedData->mutex);
    }

    if (xSemaphoreTake(sharedData->i2cMutex, pdMS_TO_TICKS(100)) == pdTRUE)
    {
        lcd.clear();
        lcd.setCursor(0, 0); lcd.print("STATE: WARNING!");
        lcd.setCursor(0, 1); lcd.print("T:"); lcd.print(localTemp, 1);
        lcd.print("C H:"); lcd.print(localHum, 0); lcd.print("%");
        xSemaphoreGive(sharedData->i2cMutex);
    }
}
\end{lstlisting}

Riêng trạng thái \texttt{CRITICAL}, nhóm bổ sung hiệu ứng nhấp nháy đèn nền LCD (backlight blink) để tăng khả năng cảnh báo trực quan cho người dùng.

\subsubsection*{Điểm nhấn kỹ thuật của Task 3}

\begin{itemize}
    \item \textbf{Đồng bộ đa tầng}: dùng cả mutex dữ liệu và mutex I2C để xử lý đồng thời hai loại xung đột khác nhau.
    \item \textbf{Tách producer-consumer}: task cảm biến chỉ phát tín hiệu trạng thái, task LCD chịu trách nhiệm hiển thị.
    \item \textbf{Thiết kế mở rộng}: cấu trúc Task 3 có thể tái sử dụng cho Task 4 (web dashboard) và Task 6 (publish cloud) nhờ vùng dữ liệu chia sẻ chuẩn hóa.
    \item \textbf{Ổn định thời gian chạy}: các nhánh lỗi (đọc DHT20 thất bại hoặc không lấy được mutex) đều có timeout và retry, giúp tránh treo hệ thống.
\end{itemize}

\subsubsection*{Tổng kết Task 3}

Task 3 đã đáp ứng các yêu cầu bắt buộc của đề bài: có cơ chế phát semaphore theo điều kiện đo, có ít nhất 3 trạng thái hiển thị trên LCD, và dùng mô hình dữ liệu chia sẻ có bảo vệ mutex thay cho cách dùng biến toàn cục rời rạc. Kết quả là hệ thống chạy ổn định trong môi trường đa task và tạo nền tảng cho các task mở rộng ở các chương tiếp theo.

\subsection{Task 4 - Webserver ở chế độ Access Point}

\subsubsection*{Mục tiêu và phạm vi}

Task 4 được triển khai nhằm đáp ứng các yêu cầu của đề bài về giao diện web điều khiển thiết bị trong chế độ Access Point. Nhóm hiện thực theo kiến trúc:
\begin{itemize}
    \item ESP32-S3 tự phát WiFi AP để người dùng truy cập trực tiếp.
    \item Dashboard hiển thị dữ liệu nhiệt độ/độ ẩm realtime.
    \item Dashboard có hai cụm điều khiển: NeoPixel RGB và quạt PWM.
    \item Trang \texttt{/settings} cho phép cấu hình STA để chuyển sang chế độ AP+STA.
\end{itemize}

\subsubsection*{Kiến trúc task và mô hình dữ liệu}

Về mặt thiết kế, Task 4 gồm hai task chính:
\begin{itemize}
    \item \texttt{actuatorTask}: điều khiển phần cứng NeoPixel và quạt theo trạng thái trong \texttt{SharedData}.
    \item \texttt{webServerTask}: chạy web server, xử lý route HTTP và cập nhật trạng thái điều khiển.
\end{itemize}

\begin{lstlisting}[language=c, caption=Khối tạo task Task 4 trong setup (trích main.cpp)]
// xTaskCreate(actuatorTask,   "Actuators",   4096, projectSharedData, 1, NULL);
xTaskCreate(webServerTask,  "WebServer",   8192, projectSharedData, 1, NULL);
\end{lstlisting}

Trong phiên bản hiện tại phục vụ kiểm thử MQTT mở rộng, nhóm đang giữ \texttt{webServerTask} hoạt động và tạm comment một số task khác (bao gồm \texttt{actuatorTask}) để cô lập luồng test. Khi demo đầy đủ Task 4, chỉ cần bật lại các lệnh \texttt{xTaskCreate(...)} tương ứng.

Để đảm bảo thread-safety khi web callback và task phần cứng truy cập cùng dữ liệu, nhóm dùng:
\begin{itemize}
    \item \texttt{mutex}: bảo vệ dữ liệu cảm biến \texttt{temperature/humidity}.

\subsection{Task mở rộng - Giao tiếp 2 thiết bị qua Local MQTT Broker}

\subsubsection*{Mục tiêu mở rộng}

Nhóm mở rộng hệ thống theo hướng giao tiếp hai vi điều khiển qua local MQTT broker (Mosquitto):
\begin{itemize}
    \item Thiết bị A (YoloUNO ESP32-S3): đọc cảm biến PIR và publish sự kiện chuyển động.
    \item Thiết bị B (ESP32-S3 DevKitC-1): subscribe sự kiện, kích hoạt buzzer 5V cảnh báo.
\end{itemize}

Mục tiêu của phần mở rộng là chứng minh khả năng liên động liên thiết bị theo mô hình publish/subscribe trong cùng mạng LAN.

\subsubsection*{Kiến trúc tổng thể và broker cục bộ}

Mosquitto được chạy trên máy tính trong mạng nội bộ, mở cổng 1883. Cả hai ESP32 kết nối broker cùng địa chỉ IP LAN (ví dụ \texttt{192.168.1.5}).

Luồng dữ liệu:
\begin{enumerate}
    \item Device A phát hiện chuyển động từ PIR.
    \item Device A publish JSON lên topic sự kiện.
    \item Device B subscribe topic này, nhận bản tin và kích hoạt buzzer.
    \item Device B publish ACK trạng thái buzzer để Device A/log theo dõi.
\end{enumerate}

\subsubsection*{Device A: Task PIR + MQTT publisher}

Trên project chính, nhóm tạo \texttt{pirMqttTask} với các cơ chế:
\begin{itemize}
    \item Chờ STA sẵn sàng (\texttt{staConnected}) trước khi kết nối MQTT.
    \item Tự reconnect MQTT mỗi 5 giây khi mất kết nối.
    \item Đọc PIR theo chu kỳ 200ms, phát hiện cạnh lên (LOW $\rightarrow$ HIGH).
    \item Dùng cooldown để chống spam bản tin liên tiếp.
\end{itemize}

\begin{lstlisting}[language=c, caption=Phát hiện cạnh lên PIR và kiểm soát cooldown (trích pir\_mqtt.cpp)]
bool currentPir = digitalRead(PIR_PIN) == HIGH;
if (currentPir && !lastPirState) {
    uint32_t now = millis();
    if (now - lastPublishTime >= PIR_COOLDOWN_MS) {
        publishMotionEvent(true);
        lastPublishTime = now;
    }
}
lastPirState = currentPir;
\end{lstlisting}

\subsubsection*{Device B: MQTT subscriber + buzzer driver}

Trên project \texttt{ESP32-S3-DevKit-C1\_Src}, nhóm chia thành ba task:
\begin{itemize}
    \item \texttt{mqttTask}: quản lý kết nối WiFi/MQTT và xử lý callback.
    \item \texttt{buzzerTask}: bật/tắt buzzer theo cờ trigger, publish ACK.
    \item \texttt{blinkLedTask}: nháy LED onboard để báo board đang hoạt động.
\end{itemize}

Chiến lược xử lý event chồng lấp: nếu buzzer đang kêu thì bỏ qua trigger mới để tránh cảnh báo kéo dài vô hạn.

\subsubsection*{Kết nối phần cứng của phần mở rộng}

\textbf{PIR (Device A):}
\begin{itemize}
    \item Chân tín hiệu PIR nối vào GPIO cấu hình trong \texttt{PIR\_PIN} (mã hiện tại dùng GPIO8).
    \item Cấu hình \texttt{INPUT\_PULLDOWN} để tránh trạng thái floating.
\end{itemize}

\textbf{Buzzer 5V (Device B):}
\begin{itemize}
    \item Không nối trực tiếp buzzer 5V vào GPIO ESP32.
    \item Dùng transistor/MOSFET driver, nối mass chung giữa nguồn 5V và ESP32.
    \item GPIO chỉ đóng vai trò tín hiệu điều khiển transistor.
\end{itemize}

\subsubsection*{Quy trình kiểm thử end-to-end}

\begin{enumerate}
    \item Chạy Mosquitto local broker và xác nhận cổng 1883 hoạt động.
    \item Nạp firmware cho Device A và Device B.
    \item Mở serial monitor cả hai board để quan sát trạng thái WiFi/MQTT.
    \item Tạo chuyển động trước PIR.
    \item Kiểm tra Device A publish event, Device B nhận event và phát còi.
    \item Kiểm tra bản tin ACK của buzzer được publish ngược lại.
\end{enumerate}

\subsubsection*{Giá trị của phần mở rộng}

Tính năng mở rộng này giúp hệ thống vượt khỏi mô hình đơn thiết bị, thể hiện năng lực:
\begin{itemize}
    \item giao tiếp liên node theo chuẩn MQTT,
    \item triển khai kiến trúc event-driven trong LAN,
    \item và thiết kế hệ cảnh báo tách rời cảm biến/chấp hành để dễ mở rộng thành mạng IoT nhiều node.
\end{itemize}

\paragraph{Tiền xử lý -- MinMax Scaling}
Đặc trưng đầu vào được chuẩn hóa về $[0, 1]$ bằng MinMax Scaler, chỉ khớp (fit) trên tập huấn luyện và áp dụng nhất quán trên tập kiểm thử lẫn firmware MCU:

\begin{equation}
x_{\text{scaled}} = \frac{x_{\text{raw}} - x_{\min}}{x_{\max} - x_{\min}}, \quad \text{giới hạn trong } [0, 1]
\label{eq:minmax}
\end{equation}

Tham số scaler được lưu vào \texttt{output/scaler\_params.json} và nhúng trực tiếp vào firmware dưới dạng hằng số:

\subsubsection*{Triển khai suy luận trên ESP32-S3}

\paragraph{Kiến trúc tích hợp firmware}
Mảng C \texttt{model\_tflite[]} từ \texttt{model\_data.h} được nhúng vào bộ nhớ Flash (phân đoạn \texttt{.rodata}), không cần hệ thống file. TFLM được cấu hình trong cặp file \texttt{tinyML.h/.cpp}. Tensor arena -- bộ đệm tĩnh cấp phát trong SRAM để chứa các activation trung gian -- có kích thước 8\,KB:

\begin{lstlisting}[language=c, caption=Khai báo tensor arena và khởi tạo TFLM (trích tinyML.cpp)]
constexpr int kTensorArenaSize = 8 * 1024; // 8 KB -- du cho mang Dense 3 tang int8

alignas(16) uint8_t tensor_arena[kTensorArenaSize];

bool initTinyML() {
    error_reporter = tflite::GetMicroErrorReporter();
    model = tflite::GetModel(model_tflite);
    // Kiem tra phien ban schema
    static tflite::AllOpsResolver resolver;
    static tflite::MicroInterpreter static_interpreter(
        model, resolver, tensor_arena, kTensorArenaSize);
    interpreter = &static_interpreter;
    if (interpreter->AllocateTensors() != kTfLiteOk) return false;
    input  = interpreter->input(0);
    output = interpreter->output(0);
    return true;
}
\end{lstlisting}

\paragraph{Đồng bộ hóa với task cảm biến qua semaphore}
Task \texttt{tinyMLTask} (stack 6\,144 bytes, ưu tiên 1) đồng bộ với task \texttt{temp\_humi} qua binary semaphore \texttt{semDataReady}. Sau mỗi lần đọc DHT20 thành công, \texttt{temp\_humi} gọi \texttt{xSemaphoreGive(semDataReady)}; \texttt{tinyMLTask} chặn tại \texttt{xSemaphoreTake(semDataReady, portMAX\_DELAY)} và chỉ tiến hành suy luận khi có dữ liệu mới:

\begin{lstlisting}[language=c, caption=Vòng lặp suy luận TinyML với đồng bộ semaphore (trích tinyML.cpp)]
while (1) {
    // Cho tin hieu du lieu moi tu task cam bien
    if (xSemaphoreTake(sharedData->semDataReady, portMAX_DELAY) == pdTRUE) {
        // Lay gia tri do qua mutex
        if (xSemaphoreTake(sharedData->mutex, pdMS_TO_TICKS(100)) == pdTRUE) {
            curr_t = sharedData->avg_temp;
            curr_h = sharedData->avg_humi;
            xSemaphoreGive(sharedData->mutex);
        }
        // Chuan hoa dau vao (MinMax)
        scale_inputs(curr_t, curr_h, &t_scaled, &h_scaled);
        // Luong tu hoa float -> int8
        input->data.int8[0] = quantize_input(t_scaled, in_scale, in_zero);
        input->data.int8[1] = quantize_input(h_scaled, in_scale, in_zero);
        // Suy luan
        interpreter->Invoke();
        // Phan luong tu hoa int8 -> float, lay argmax
        // Ghi nhan vao sharedData->mlLabel qua mutex
    }
}
\end{lstlisting}

Cơ chế \texttt{portMAX\_DELAY} đảm bảo \texttt{tinyMLTask} không tiêu tốn CPU khi không có dữ liệu mới -- phù hợp với chu kỳ đo 5\,000\,ms của DHT20.

\paragraph{Lượng tử hóa và phản lượng tử hóa trên MCU}
TFLite int8 sử dụng phép biến đổi affine với tham số scale và zero\_point đọc trực tiếp từ metadata tensor:

\begin{lstlisting}[language=c, caption=Hàm lượng tử hóa và phản lượng tử hóa trên MCU (trích tinyML.cpp)]
// Luong tu hoa: float -> int8
static int8_t quantize_input(float value, float scale, int zero_point) {
    const float q = value / scale + (float)zero_point;
    int32_t qi = (int32_t)lroundf(q);
    if (qi < -128) qi = -128;
    if (qi >  127) qi =  127;
    return (int8_t)qi;
}
// Phan luong tu hoa: int8 -> float xac suat
static float dequantize_output(int8_t value, float scale, int zero_point) {
    return (float)(value - zero_point) * scale;
}
\end{lstlisting}

Nhãn dự đoán (\texttt{mlLabel}) được ghi vào \texttt{SharedData} qua \texttt{mutex} và có thể được sử dụng bởi các task khác như màn hình LCD hoặc NeoPixel để hiển thị trạng thái phân loại.

\subsubsection*{Thảo luận và đánh giá kết quả}

\paragraph{Phân tích kết quả đánh giá}

\begin{itemize}
    \item \textbf{Lớp nào sai nhiều nhất?} Không có lớp nào bị phân loại sai trên cả ba bộ đánh giá (float32, int8 trên PC, kịch bản biên giới). Điều này cho thấy mô hình Dense với 291 tham số đủ năng lực phân tách ba lớp dữ liệu tổng hợp có cấu trúc rõ ràng.

    \item \textbf{Lỗi có tập trung tại biên ngưỡng không?} Không. Kiểm thử 6 kịch bản biên giới trong \texttt{mock\_scenarios.py} đều cho kết quả đúng, kể cả các điểm tại giá trị clip cực đại (T=$-10$°C, T=60°C, H=0\%, H=100\%).

    \item \textbf{Độ chính xác trên thiết bị thấp hơn offline không?} Không quan sát thấy sự suy giảm. Mô hình int8 cho độ chính xác tương đương mô hình float32 vì khoảng cách giữa các lớp trong dữ liệu tổng hợp đủ lớn để sai số lượng tử hóa (±1 LSB) không gây phân loại sai.

    \item \textbf{Dự đoán có bất ổn theo thời gian không?} Không. Mỗi lần suy luận độc lập, không có trạng thái thời gian. Độ tin cậy đầu ra cao (ví dụ P(Normal) $> 0.99$ với T=26,5°C/H=60\%).
\end{itemize}

\paragraph{Hạn chế}

\begin{itemize}
    \item \textbf{Khoảng cách synthetic-to-real}: Dữ liệu tổng hợp có phân phối lý tưởng, không phản ánh nhiễu đo thực tế của DHT20 ($\pm$0,5°C, $\pm$3\%RH). Độ chính xác thực tế trên dữ liệu cảm biến thật dự kiến sẽ thấp hơn 100\%.
    \item \textbf{Không có mô hình thời gian}: Mỗi lần suy luận xử lý độc lập một điểm đo -- các bất thường phát triển dần (drift) có thể bị bỏ sót.
    \item \textbf{Hằng số scaler cứng}: \texttt{kDataMin/kDataMax} là hằng số compile-time; khi triển khai môi trường thực cần cập nhật lại và biên dịch lại firmware.
\end{itemize}

\subsubsection*{Tổng kết Task 5}

Task 5 đã hoàn thành một pipeline TinyML khép kín và đúng yêu cầu đề bài: có mô tả tập dữ liệu và quy trình gán nhãn; mô hình được triển khai trực tiếp trên vi điều khiển; có đánh giá độ chính xác trên cả PC (float32 và int8) lẫn kịch bản thực; có thảo luận về nguyên nhân kết quả và các giới hạn khi áp dụng thực tế. Cơ chế đồng bộ \texttt{semDataReady} đảm bảo task TinyML hoạt động hiệu quả trong môi trường đa task của FreeRTOS, không gây tải dư thừa lên CPU.
